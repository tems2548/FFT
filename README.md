# ESP32-S3 Live FFT Spectrum Analyzer

A two-part tool for turning an ESP32-S3 into a USB-tethered oscilloscope / spectrum
analyzer / signal quality meter:

- **`src/main.c`** — ESP-IDF firmware that samples an analog input via the ADC's
  continuous (DMA) mode, decimates it to a manageable rate, and streams calibrated
  voltage samples to a PC over the same USB-UART used for flashing/logging.
- **`src/FFT.py`** — A PyQt6 + pyqtgraph desktop app that receives that stream (or
  generates a synthetic test signal instead) and renders a live multi-panel
  time-domain / frequency-domain / spectrogram / phase display, plus a full panel of
  standard signal-quality measurements (SNR, SINAD, ENOB, THD, SFDR, harmonics, ...).

Also works with **no hardware at all** — `FFT.py` doubles as a self-contained FFT test
bench with synthetic waveforms (sine/chirp/square/sawtooth/demo), useful for learning
FFT/DSP concepts or exercising the visualizer on its own.

## Hardware

- Board: ESP32-S3 (tested on an `esp32-s3-devkitm-1`)
- Signal input: GPIO7 (`ADC_CHANNEL_6`, ADC unit 1), 0–3.3V range (`ADC_ATTEN_DB_12`)
- Connection: a single USB cable (used for both flashing and the live data stream)

If your circuit's signal is centered on a mid-supply DC bias (e.g. an op-amp front end
biased at 1.65V) rather than swinging around 0V, that's expected — the Python side
AC-couples it in software and reports the removed bias separately (see
[Features](#features) below).

## Architecture

```
 [Analog signal] -> GPIO7 -> ESP32-S3 ADC (continuous/DMA mode, ~80 kHz raw)
                                   |
                       decimate (average) + calibrate to mV
                                   |
                    frame as META/DATA packets + CRC-16
                                   |
                    UART0 @ 3,000,000 baud, same port as USB flashing
                                   |
                              [USB cable]
                                   |
                   FFT.py: SerialReader (background thread)
                                   |
                    PyQt6 / pyqtgraph live multi-panel UI
```

## Wire protocol

Packets are framed with a 4-byte ASCII magic word so the Python parser can resync past
any stray `ESP_LOG` text sharing the same UART, and are self-correcting: a corrupted
packet is discarded (via CRC) without breaking sync on the next valid one.

All multi-byte fields are little-endian (native ESP32 byte order).

| Packet | Layout | Notes |
|---|---|---|
| `META` | `"META"` + `uint32 sample_rate_hz` + `uint16 crc16` | Sent once at boot and re-sent every 1s, so a client connecting after the board has already been running still learns the rate within a few seconds. |
| `DATA` | `"DATA"` + `uint16 count` + `count × int16 millivolts` + `uint16 crc16` | Batched every 128 decimated samples. |

CRC is **CRC-16/CCITT-FALSE** (poly `0x1021`, init `0xFFFF`) computed over everything
in the packet after the magic word. It's implemented independently on both sides
(`crc16_ccitt_update()` in `main.c`, `crc16_ccitt()` in `FFT.py`) and verified against
the standard test vector (`CRC(b"123456789") == 0x29B1`).

### Why decimation + rate self-measurement

The ADC continuous driver's actual conversion rate doesn't reliably match the
`sample_freq_hz` requested of it (observed both ~2× over and ~2× under, depending on
the value). Rather than trust the configured rate, the firmware measures the real raw
rate for ~1s at boot (`measure_actual_raw_sample_rate`) and derives a decimation factor
from that measurement to land close to `TARGET_OUTPUT_RATE_HZ` (currently 80000, i.e.
close to unthrottled) — so the frequency axis Python builds from the `META` packet is
accurate regardless of what the hardware actually does. Lower `TARGET_OUTPUT_RATE_HZ`
in `main.c` if you only care about lower-frequency signals; the decimation averaging
also acts as a (weak, boxcar) anti-alias filter and reduces noise.

## Firmware setup

Built with [PlatformIO](https://platformio.org/) (ESP-IDF framework).

```
pio run                    # build
pio run --target upload    # flash
pio device monitor         # view logs (3,000,000 baud, see below)
```

### Console UART baud (3,000,000)

The console UART baud is raised from ESP-IDF's default 115200 to 3,000,000 in
`sdkconfig.defaults`, because at the default rate (~11,500 bytes/sec of real
throughput) the data stream would need to be decimated down to ~4-5 kHz to fit — at
3 Mbaud (~300,000 bytes/sec) it comfortably carries the near-unthrottled ~80 kHz
stream instead. `FFT.py`'s `--baud` (or the port-picker dialog's baud field) must
match this.

This setting was non-trivial to make stick — worth knowing if you ever need to change
it again:
- Editing the generated `sdkconfig.esp32-s3-devkitm-1` directly doesn't persist; it's
  regenerated by the build and any manual edits are silently discarded.
- `CONFIG_ESP_CONSOLE_UART_BAUDRATE`'s Kconfig *prompt* is only active under the
  "Custom UART" choice (`ESP_CONSOLE_UART_CUSTOM`) — under the default "UART0" choice,
  the baud value is ignored no matter what any config file says.
- The fix, now in `sdkconfig.defaults`: select `CONFIG_ESP_CONSOLE_UART_CUSTOM=y` +
  `CONFIG_ESP_CONSOLE_UART_CUSTOM_NUM_0=y` (which still defaults to the same UART0 /
  GPIO43(TX) / GPIO44(RX) pins already wired to the board's USB-serial bridge — nothing
  physically changes) alongside the baud itself. If you change this file, delete the
  generated `sdkconfig.esp32-s3-devkitm-1` and do a clean build
  (`pio run --target clean && pio run`) so it's regenerated from the new defaults.

## Python visualizer setup

Requires Python 3 with:

```
pip install numpy pyqtgraph PyQt6 pyserial
```

Run against live hardware:

```
python src/FFT.py --serial              # pick the port from a GUI list
python src/FFT.py --serial COM5         # or connect directly
python src/FFT.py --serial /dev/ttyUSB0 --baud 3000000
```

Run standalone with a synthetic test signal (no hardware needed):

```
python src/FFT.py
python src/FFT.py --wave chirp --freq 20 --freq2 80
python src/FFT.py --wave sine --freq 440 --samplerate 44100
```

### CLI reference

| Flag | Default | Meaning |
|---|---|---|
| `--serial [PORT]` | off (synthetic mode) | Live ADC mode. Omit the port to open a GUI picker. |
| `--baud` | `3000000` | Serial baud for `--serial`; must match the firmware's `CONFIG_ESP_CONSOLE_UART_BAUDRATE`. |
| `--wave` | `demo` | Synthetic waveform: `demo`, `sine`, `chirp`, `square`, `sawtooth`, `noise`. |
| `--freq`, `--freq2`, `--sweep-period` | `10`, `60`, `8.0` | Synthetic signal parameters (freq2/sweep-period are for `chirp`). |
| `--samplerate` | `2000.0` | Synthetic mode's sample rate (Hz); ignored in live mode (learned from `META`). |
| `--window` | `2048` | FFT / display window size, in samples. Raise this for live captures at high sample rates — see note below. |
| `--fft-window` | `Hann` | FFT window function: `Hann`, `Hamming`, `Blackman`, `Blackman-Harris`, `Flat-top`, `Rectangular`. Also switchable live. |
| `--averaging` | `70.0` | Spectrum trace averaging, 0 (instant) to ~99 (heavy smoothing). Also switchable live. |
| `--noise` | `0.03` | Synthetic mode's initial noise level. |
| `--fps` | `30` | UI update rate. |

Everything else (window function, averaging, log axis, peak hold, cursors, pause,
wave/freq/noise for synthetic mode) is also adjustable live from the sidebar.

### A note on window size vs. sample rate

At a high output rate with a small FFT window, consecutive frames can end up
non-overlapping (more new samples arrive per frame than the window even holds), which
looks jumpy and gives coarse frequency resolution. If your live capture looks unstable,
raise `--window` (e.g. `--window 16384`) and/or the sidebar's Averaging slider.

## Features

**Time domain** — scrolling waveform, AC-coupled (DC bias auto-subtracted) in live
mode with the removed bias shown separately in the readings panel.

**Frequency domain** — live spectrum with:
- Selectable FFT window function (trades frequency resolution against
  amplitude/SFDR accuracy)
- Adjustable trace averaging (power-domain exponential smoothing — statistically
  correct, unlike averaging dB directly)
- Optional log-frequency X axis
- Peak-hold overlay (dashed trace, tracks the highest level ever seen per bin —
  catches intermittent spurs the live trace alone would miss), with a reset button
- Hover crosshair readout
- Fundamental peak and second-highest independent peak markers (for two-tone /
  interference analysis — the "2nd peak" is *not* restricted to harmonics of the
  fundamental)
- Dual draggable delta cursors — measure Δfrequency / ΔdB between any two points

**Phase spectrum** — instantaneous phase (not averaged — see the code comment on why
vector-averaging a free-running capture would be actively wrong), with bins more than
10dB below the noise floor blanked out rather than plotted (raw phase is meaningless
noise there).

**Spectrogram** — scrolling waterfall (frequency × time-ago × magnitude color).

**Noise floor & SNR trend** — rolling history of both over time.

**Readings panel**, organized into sections:
- *Signal*: peak freq/dB, 2nd-highest peak, DC bias, peak-to-peak amplitude, RMS,
  crest factor
- *Harmonics*: 2nd through 5th, each with sub-bin-accurate frequency/level
- *Signal Quality*: SNR, SINAD, ENOB (derived from SINAD), THD (reports `n/a` rather
  than a misleading `0%` once the fundamental is high enough that even the 2nd
  harmonic exceeds Nyquist — a hard physical limit, not a bug), SFDR

**Dual interactive delta cursors** — draggable on both the time and frequency plots,
with a live readout of each cursor's position and the delta between them (ΔT is also
shown as its reciprocal frequency, handy for reading a period directly off the trace).

**Pause / Freeze** — stops the display cold for close inspection; the live serial
queue keeps draining in the background (samples discarded, not buffered) so it doesn't
grow unbounded while paused. "Save CSV" naturally captures whatever was on screen when
paused.

**Link health** (live mode only) — packet-level CRC pass/fail counters, shown in the
sidebar and turning red the moment any corruption is detected.

**CSV export** — snapshots the full time/frequency arrays plus every computed metric.

## Known limitations

- **THD above ~Nyquist/2 fundamentals**: correctly reported as unmeasurable (see
  above) rather than silently wrong.
- **Single channel**: only one ADC channel (GPIO7) is sampled; no multi-channel /
  Lissajous support yet.
- **No trigger mode**: capture is free-running; there's no edge/level-triggered
  single-shot capture (Pause + cursors are the closest substitute today).
- **No auto-reconnect**: if the board resets or the USB link drops mid-session, the
  display just freezes with no automatic retry (the Link health counters will show
  the packet flow having stopped).
- **Spectrogram frequency axis is always linear** — the log-frequency toggle only
  applies to the Frequency domain and Phase spectrum plots, since `ImageItem` doesn't
  support a non-linear axis without resampling the underlying image data.
